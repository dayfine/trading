- [info] **Working-tree clobber cycle.** Every subagent I spawned in this run did `git fetch origin && git reset --hard origin/<branch>` at session start per `.claude/agents/*.md` setup blocks. Each reset clobbered the orchestrator's uncommitted reconciliation edits to `dev/status/_index.md`, `dev/status/sector-data.md`, and `dev/status/cleanup.md`. Mitigation this run: re-applied edits twice, then committed to `ops/daily-2026-04-19-run4` so further subagents wouldn't clobber (and would merely switch branches). Follow-up: Step 5.5's reconciliation edits should move to happen on a dedicated orchestrator branch created at the top of the run, *before* any subagent dispatch, so edits are already-committed when subagents checkout. Worth a doc PR into `.claude/agents/lead-orchestrator.md` §Step 5.5 (pre-dispatch vs post-dispatch split).

- [info] **Stray commits from sibling-agent branches.** The harness-maintainer agent reported catching "two stray commits from sibling agent's branch" in its initial push and had to force-push after cherry-picking onto a clean checkout. This is the same worktree-isolation hazard documented in `.claude/rules/worktree-isolation.md` — but it keeps recurring because multiple parallel subagents share the same GHA working tree. In the container, our "parallelism" is effectively sequential (second agent starts on the residual state of the first). Surfacing as an `[info]` rather than `[critical]` because the harness-maintainer detected it and self-corrected, and the final PR #448 is clean.

- [info] **Status_file_integrity dune rule does not depend on status files.** `trading/devtools/checks/dune` line 93-97 declares `(deps _check_lib.sh)` but not the actual `dev/status/*.md` files the script reads. `actions/cache@v4` key in `.github/workflows/ci.yml` hashes `.opam/.ml/.mli/dune` files, so when status files change CI restores a stale cached "OK" for the rule. Local `dune runtest` also returns cached success until `--force`. This is harmless today (the script itself is deterministic and the cache is just an output cache) but worth fixing: either (a) add the status files as explicit deps on the rule, or (b) have the rule depend on a manifest file that ratchets when any `dev/status/*.md` changes. Harness-maintainer ticket candidate for run-5.

- [info] **3c stacked dispatch path went non-obvious.** I dispatched feat-backtest for 3c stacked on `feat/backtest-tiered-loader-3b-ii-summary-tier` (PR #445, which was then unmerged). During the agent's session, the human merged #445, so the agent rebased onto `origin/main` and opened #447 with `base: main` instead of stacking. This is the right outcome — stacking on an unmerged PR adds no value once its root merges — but the dispatch message assumed "stacked on #445" so the status-file and summary wording needed cleanup post-facto. Lesson: the Step 1.5 stack-eligibility logic should treat "root PR may merge mid-session" as a normal case and let the agent rebase to main if the root goes away.
