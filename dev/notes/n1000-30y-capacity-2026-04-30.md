# N=1000×30y capacity validation (2026-04-30)

First measured-peak-RSS run at the long-horizon cell: **N=1,000 symbols
× 30 years (1996-01-02 → 2025-12-31)** on a survivorship-biased
sector-classified cohort. This run is a **CAPACITY VALIDATION ONLY**;
strategy correctness over a 30y window is contaminated by survivorship
bias (issue #696, see
`dev/notes/historical-universe-membership-2026-04-30.md`). The metrics
reported below are NOT representative of live-tradeable returns.

## Setup

- **Host:** Darwin 25.4.0 (macOS), 32 GB physical RAM. Docker Desktop
  VM ceiling is **7.75 GiB** (default; not user-resized for this run).
  This is identical in size to the GHA `ubuntu-latest` 8 GB ceiling —
  so the run is in fact a *GHA-equivalent* capacity test, not a
  full-host capacity test.
- **Container:** `trading-1-dev`, opam env, OCaml.
- **Build SHA:** `374b575b` (post-#694 — soft cash floor on shorts;
  feature-G3 just landed; G4 still in flight).
- **GC:** `OCAMLRUNPARAM=o=60,s=512k` (tuned). The first run at
  default GC peaked at **6.96 GB** RSS and was kernel-OOM-killed by
  the 7.75 GiB VM ceiling at ~3 minutes wall (cgroup memory.events
  recorded oom_kill 1). GC tuning was applied for the successful run
  reported below; tuning shaved ~10 % off peak RSS and let the run
  complete cleanly.
- **Cost-model projection** (post-engine-pool, from
  `dev/notes/panels-rss-matrix-post-engine-pool-2026-04-28.md`):

  ```
  RSS ≈ 67 + 3.94·N + 0.19·N·(T − 1)
       = 67 + 3,940 + 0.19·1,000·29
       ≈ 9,517 MB (~9.3 GB)
  ```

  Cost-model worst-case projection 9.3 GB; measured (GC-tuned)
  6.18 GB; projection × 0.65 — between the 0.52× factor measured at
  N=1000×10y and the worst-case projection. The 30y horizon γ does
  not blow up dramatically.
- **N:** 1,000 trading symbols + index + sector ETFs = 1,015 total
  loaded.
- **T:** 30 years (1996-01-02 to 2025-12-31), 7,977 trading days
  (warmup from 1995-06-06).
- **Universe:** `universes/broad-1000-30y.sexp` — generated by
  `dev/scripts/curate_30y_universe.sh`.
  - All 1,000 symbols have `data_start_date <= 1996-01-01` (verified).
  - All 1,000 symbols carry GICS sector classification from
    `data/sectors.csv`.
  - **Composition:** 305 SP500-30y + 695 alphabetic backfill from the
    866 non-SP500 sectors-30y candidates.
  - **Start-year buckets** (number of symbols starting in each year,
    selected examples): 1962×25, 1972×16, 1973×57, 1980×157,
    1984×169, 1985×27, 1986×48, 1987×45, 1988×29, 1989×24, 1990×19,
    1991×41, 1992×55, 1993×69, 1994×78, 1995×89. Earliest 1962-01-02,
    latest 1995-12-21 — entire cohort has full 30y window coverage.
  - **End-date coverage:** 396 symbols end 2025-05-16, 604 symbols
    end 2026-04-10. Period requested 1996-01-02 → 2025-12-31; trailing
    bars 2025-05-17 → 2025-12-31 are missing for ~40 % of the cohort.
    The bar loader handles partial trailing coverage gracefully —
    the run completed cleanly, no errors during simulation.
  - **Sector distribution:** Financials 208, Industrials 166,
    Information Technology 112, Consumer Discretionary 110, Health
    Care 105, Materials 73, Consumer Staples 60, Real Estate 57,
    Utilities 48, Energy 44, Communication Services 17. Heavy
    financial/industrial weighting reflects what survived 30 years
    on this cohort selection method.

- **Scenario:**
  `goldens-broad/sp500-30y-capacity-1996.sexp` — `enable_short_side =
  false`, `universe_cap = 1000`. Mirror of #682's mitigation: G4 not
  yet merged at scenario-creation time, so long-only is the safe
  default. NOT a release-gate cell — explicitly marked
  `perf-tier: capacity-only` in the sexp header.

- **Command:**

  ```sh

    'mkdir -p /tmp/n1000-30y-cell && \
  ```

### Comparison with N=1000×10y baseline

(Reference: `dev/notes/n1000-decade-rss-2026-04-29.md`,
`goldens-broad/decade-2014-2023`.)

| Metric             | N=1000×10y       | N=1000×30y       | Ratio  |
|--------------------|-----------------:|-----------------:|-------:|
| Peak RSS           | 2,945 MB         | 6,328 MB         | 2.15×  |
| Wall (real)        | 4:26.65          | 31:58.63         | 7.20×  |
| User CPU           | 246.98 s         | 1,834.79 s       | 7.43×  |
| Trading days       | 2,757            | 7,977            | 2.89×  |
| Symbols loaded     | 1,015            | 1,015            | 1.00×  |
| Major page faults  | 74               | 279,831          | 3,781× |
| Minor page faults  | 694,118          | 13,010,631       | 18.7×  |

**Wall scaled super-linearly** (2.89× more days, 7.20× more wall) —
the 30y run is paying ~2.5× per-day tax compared to 10y. Two likely
sources:

1. **VM-ceiling pressure.** 6.18 GB / 7.75 GB ceiling = **80 %
   utilization**. Major page faults jumped 3,781× — the engine spent
   significant wall thrashing the page cache as the OS swapped pages
   to keep within the cgroup limit. This is exactly what the task
   anticipated would happen on a 32 GB host (which Docker Desktop
   restricts to 7.75 GiB by default).
2. **Per-tick state grows with simulation history.** Stops, panel
   indicators, and per-symbol state machines all carry data
   proportional to elapsed simulation. γ in the cost model captures
   memory; wall has its own γ that the cost model doesn't fit.
3. **GC pressure.** The `o=60,s=512k` tuning trades GC frequency for
   peak heap size — more frequent collections during a bigger panel
   space adds CPU.

**Peak RSS scaled 2.15×** for 2.89× more days × 1× symbols — the
β=3.94 MB/symbol coefficient is dominant; γ is small (0.19 MB/sym/yr).
Confirms the cost-model shape but at a smaller magnitude than worst
case.

### Cohort dead-zone — survivorship-bias diagnostic

**All 8 trades were entered on 1996-01-06** (the very first
post-warmup day where the screener can return Stage-2 candidates).
**No trade was entered on any subsequent day in the 30-year run.**
The last trade exited 1997-03-20; the strategy held zero positions
for ~28.7 years.

This is a textbook survivorship-cohort artefact. With every symbol in
the universe being a 30y+ survivor, almost all of them are continuously
in or near Stage 2 by 1996, with no clear breakout-from-Stage-1
transitions remaining (those would have happened *before* the cohort
selection window). The screener's stage classifier and resistance
detector therefore find no Stage-1 → Stage-2 transitions to enter on
in the 1996-01-07 → 2025-12-31 window. The few stage-2-now-but-
recently-1 candidates that did exist on 1996-01-06 all stopped out
quickly (tight initial stops; this was post-#682 mitigation).

Confirms the survivorship-bias-warning prediction: a 30y backtest
on a 30y-survivor cohort is **structurally degenerate** as a strategy
test. Any strategy validation at this horizon must use point-in-time
membership (P5 of #696); without it, the result is a no-op.

## Observations

- **Run completed cleanly.** Exit status 0, scenario PASS, no errors,
  no negative-cash days, no NaN drawdown. The engine handles the 30y
  horizon at N=1000 correctly.
- **VM ceiling pressure is real.** Major page faults 3,781× the 10y
  baseline → the run was thrashing the page cache. Wall scaled
  7.2× for 2.89× more days, dominantly because of paging.
- **OOM at default GC; OOM-free at tuned GC.** The capacity envelope
  for N=1000×30y on a 7.75 GiB VM is **only available with GC tuning**.
  Without `OCAMLRUNPARAM=o=60,s=512k`, peak RSS = 6.96 GB exceeds the
  margin and triggers kernel OOM around mid-run.
- **No determinism issue observed.** Single cell, single run; the
  multi-cell determinism issue (G6 caveat in #687) is not in scope here.
- **Macro trend dump is truncated to warmup window.** The
  `macro_trend.sexp` artefact contains only weeks 1995-06-09 to
  1996-01-05 (31 Bullish entries) — the post-warmup macro trend never
  gets persisted. This appears to be a separate issue with
  `Result_writer` / dump cadence; tracked here as a follow-up
  observation rather than blocking the capacity result.
- **Trade pattern is a survivorship-bias canary.** 8 trades, all
  entered on day 1, none after — see "Cohort dead-zone" above.
- **No split-event anomalies.** The run did not crash on any of the
  ~6,000 expected split events (per
  `dev/notes/data-availability-2026-04-29.md` §4 spot check). The
  per-tick `_apply_splits_to_positions` pathway handled the 30y span.

## What this run does NOT establish

- **Strategy quality.** The metrics above are contaminated by
  survivorship bias *plus* the cohort dead-zone artefact. A re-run
  with point-in-time SP500 membership (P5 of issue #696) would
  produce dramatically different results — historical Stage-1 → 2
  transitions would be intact for symbols that didn't yet exist or
  hadn't yet survived in 1996.
- **Non-survivor capacity envelope.** This run loads 1,015 symbols
  all with full 30y bar coverage. A point-in-time cohort would load
  N×~1.5 symbols (rolling membership over 30y) — the per-symbol RSS
  cost changes when many symbols are short-lived rather than
  full-history.
- **Full-host capacity envelope.** This run hit the 7.75 GiB Docker
  Desktop VM ceiling, *not* the 32 GB physical host. To measure the
  unconstrained envelope, the Docker Desktop VM memory needs to be
  resized in Docker Desktop's settings UI (no CLI-controllable
  override exists). Future capacity-pushing experiments at N>=2000
  or T>=30 should resize the VM first.
- **GHA `ubuntu-latest` feasibility at N=1000×30y.** Marginal: peak
  6.18 GB inside an 8 GB ceiling = 77 % utilization, with substantial
  paging. Probably runs but slowly. Daily-snapshot streaming
  (`dev/plans/daily-snapshot-streaming-2026-04-27.md`) remains the
  recommended path for any production CI use of this cell.
- **GC-untuned envelope.** Default GC OOMed mid-run at peak 6.96 GB.
  GC tuning is **required**, not optional, for this cell at the
  current VM size.

## Re-run trigger

- Docker Desktop VM resize (>=12 GiB) lands → re-run to confirm the
  unconstrained envelope (peak should drop, wall should drop ~3-5×
  due to paging elimination).
- Daily-snapshot streaming lands → re-run to verify the streaming
  envelope at N=1000×30y.
- Point-in-time historical membership filter lands (P5 of #696) →
  re-run with realistic delisting handling to record the
  strategy-validation baseline.
- A new γ-affecting feature lands (per-bar instrumentation, per-symbol
  state) → re-run as a regression check on the long-horizon cell.
