**The streaming fix worked where I expected:**

**But it didn't help where I expected:**
- `Promote_full` still climbs to ~3.6 GB. The streaming fix made no visible difference here.

## Why H7 didn't move the Promote_full needle

Promote_full does more than just call `Csv_storage.get`. After loading bars, it:
1. Builds a `Full.t = { bars; summary; metadata }` per symbol ‚Äî `bars` is `tail_days` long (default 1800).
2. Computes `summary` = `Summary_compute.compute_values` ‚Äî RS line, 30-week MA, ATR, stage heuristic. Each indicator allocates intermediate weekly aggregations.
3. Stores in the loader's per-symbol map.
4. The wrapper's `_promote_universe_to_full` then calls `_seed_from_full` per symbol, which:
   - Calls `_truncate_bars` ‚Äî `List.filter` over the loaded bars
   - Calls `Bar_history.seed` ‚Äî `existing @ new_bars` allocates a new combined list
5. For 292 symbols √ó 312 Friday cycles over the run.

Streaming the CSV READ helped phase 1 a bit (130 MB on Metadata which is just the read), but phases 2-5 are ALSO Tiered-only and allocate substantially. They're downstream of the CSV.

Specifically the smoking-gun callsite from the peak memtrace analysis was `Base.List.filter` (+773 MB Tiered overhead) ‚Äî that's `_truncate_bars` AND `_in_date_range` filtering AND `Summary_compute`'s indicator iterations, not just Csv_storage.get.

We've ruled out four hypotheses. The +95% Tiered RSS gap is **not** in:
- Bar_history's monotonic growth
- Full.t.bars retention
- AD-breadth load
- Csv_storage.get materialization

The actual carrier is **somewhere in the per-symbol Promote_full work AFTER the CSV read**. Candidates remaining:
- `Summary_compute.compute_values` intermediate state (weekly aggregations √ó indicators √ó symbols)
- `_seed_from_full` ‚Üí `_truncate_bars` (List.filter on loaded bars)
- `Bar_history.seed` ‚Üí `existing @ new_bars` (List.append churn √ó Friday cycles)
- Some Bar_loader bookkeeping per symbol (the `t.full_map` Hashtbl plus internal state)

## What now

Three options for next H-test:

1. **H8 ‚Äî Skip Summary scalar computation in Promote_full.** Add a config field that promotes to Full WITHOUT computing Summary scalars (just bars + metadata). If RSS drops substantially, the indicator computations are the carrier.

2. **H9 ‚Äî Refactor `Bar_history.seed` to share lists rather than copy.** Currently `existing @ new_bars` allocates a fresh combined list each Friday. Could mutate or use a more efficient deque. ~50 LOC change to bar_history.ml.

3. **H10 ‚Äî Memtrace-only investigation.** Now that we have memtrace working (PR #538), open the H7-verify .ctf in `memtrace_viewer` (requires `opam install memtrace_viewer` on user's host ‚Äî not currently available in our docker due to OCaml 5 conflict). The flamegraph would directly show the Promote_full callsite tree.

Recommended order: **H8 first** (cheap to test ‚Äî config toggle exists pattern). H9 is a code change that needs design care.

## Methodological note

This is the 4th disproved hypothesis. The diagnosis-fix-measure cycle has been productive but the +95% gap is genuinely elusive. Three insights from these disproofs:

1. **Memtrace's "live at peak" attribution by callsite** points correctly (CSV._build_price 2.3√ó more in Tiered) but the FIX target needs more care ‚Äî the byte attribution is for ALLOCATION sites, not RETENTION carriers. Just because callsite X allocates 2√ó more doesn't mean fixing X drops RSS.

2. **OCaml's heap doesn't shrink** ‚Äî once expanded to peak, RSS stays there. So fixing a churn source mid-run that's NOT at the peak phase doesn't help.

3. **The phase-by-phase RSS view is more diagnostic than total RSS** ‚Äî H7 dropped 130 MB on Metadata but added 75 MB on Promote_full (probably noise). Total = unchanged. Without per-phase view we'd have called this a complete null.
