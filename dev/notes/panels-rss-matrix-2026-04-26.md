# Panel-mode RSS matrix — N × T scaling on post-PR-D (2026-04-26)

Companion to the four post-Stage-4 spike notes. Answers two questions:

1. Is the memory cost **fixed** (one-time setup) or **variable** (scales
   with N × T)?
2. Where should optimization look next?

## Method

Four cells on `/tmp/data-small-302` (post-PR-D Panel mode), Friday-anchored
date ranges, `universe_cap` overrides. Wall + peak RSS via
`/usr/bin/time -v`.

## Result

| N (cap) | T (years) | Peak RSS (MB) | Wall |
|---:|---:|---:|---:|
| 50 | 1 (2019) | 342 | 0:15 |
| 50 | 6 (2015–2020) | 408 | 1:08 |
| 292 | 1 (2019) | 1,581 | 1:08 |
| 292 | 6 (2015–2020) | 1,861 | 4:39 |

## Fit

`RSS ≈ 86 + 5.12·N + 0.22·N·(T − 1)` MB

| Component | Value | Interpretation |
|---|---:|---|
| Fixed (α) | ~86 MB | Universe-independent: OHLCV + indicator panels (Bigarray, ~50 MB), runtime, fixed allocations |
| Per-symbol (β) | ~5.12 MB | Dominant cost — per-loaded-symbol structures |
| Per-symbol-per-year (γ) | ~0.22 MB | Small T-dependent term — accumulating state (Stop_log, trade history, Trace) |

**Verdict: memory is dominated by N (universe size), not T.**
- N=50 → 292 (5.84×) gives RSS 4.6× (close to linear in N).
- T=1y → 6y (6×) gives RSS only 1.18× (almost flat).

## What costs 5 MB / symbol

The **C-heap Bigarray panels** account for only ~50 MB total at full
universe (N_loaded ≈ 307, T = 1,715 days):
- 6 OHLCV panels × 307 × 1,715 × 8 bytes ≈ 25 MB
- 4 indicator panels × 307 × 1,715 × 8 bytes ≈ 17 MB
- 2 RSI scratch arrays × 307 × 1,715 × 8 bytes ≈ 8 MB

So the remaining ~1,810 MB at N=292 T=6y lives in **OCaml heap**, not
Bigarray. That's where 5 MB / symbol comes from.

Code inspection of `_screen_universe` (`weinstein_strategy.ml:266`)
reveals the structural cause:

```ocaml
let stocks = List.filter_map config.universe ~f:_analyze_ticker in    (* runs for ALL N *)
let screen_result = Screener.screen ~macro_trend ~sector_map ~stocks   (* filters AFTER *)
```

`_analyze_ticker` does the full per-symbol heavy work — `weekly_view_for`,
`Panel_callbacks.stock_analysis_callbacks_of_weekly_views` (Stage / Rs /
Volume / Resistance bundles), `Stock_analysis.analyze_with_callbacks` —
**for every loaded symbol on every Friday**, regardless of whether macro
is bearish or the symbol's sector is weak. The screener cascade
(`Screener.screen`) gates **after** all per-symbol analysis is already
done. No early exit.

Other suspects (less load-bearing):
1. `daily_bars_for` / `weekly_bars_for` fallback churn — Volume +
   Resistance still go through `Bar_reader.weekly_bars_for` for the
   `bars_for_volume_resistance` fallback path.
2. `Stop_log` + `Trading_state` accumulating per-position history.
3. `Trace` event ring retained for the whole run.
4. `Macro` AD-bar list held throughout.

## Release-gate projection

Plan target: ≤ 8 GB at N=5,000 T=10y (release-gate tier 4).

Extrapolating the fit:

`RSS(5000, 10) ≈ 86 + 5.12·5000 + 0.22·5000·9 = 86 + 25,600 + 9,900 = 35,586 MB ≈ 35 GB`

**4× over the 8 GB ceiling**, consistent with the original plan's
estimate that Stage 4 wouldn't get there alone. The 5 MB/symbol slope
needs to come down to ~1.5 MB/symbol to fit.

## Sweep-harness gap

The pre-Stage-4 sweep notes
(`dev/notes/bull-crash-sweep-2026-04-25.md`,
`dev/notes/perf-sweep-2026-04-25.md`) were generated by Python scripts
deleted in #582 (no-Python rule). The N × T matrix above was built
ad-hoc — there is no automated sweep harness in the repo today.

**Tier-1 smoke runner** (`dev/scripts/perf_tier1_smoke.sh`) covers the
`perf-tier: 1` scenarios but does not vary `universe_cap`. Tier-3 weekly
workflow is unwritten (per `dev/status/backtest-perf.md` Step 2).

A small POSIX-shell sweep harness — ~50 LOC — that reads the
`perf-sweep/*.sexp` files and runs each at `--override "((universe_cap (N)))"`
for N ∈ {10, 50, 150, 292, 1000}, captures `Maximum resident set size`
+ wall, emits CSV. Optional follow-up; the manual matrix above already
answers the immediate question.

## Recommendation

**Stage 4.5 — lazy-tier cascade.** See
`dev/plans/panels-stage045-lazy-tier-cascade-2026-04-26.md`. The
finding above (per-symbol heavy work fires for all loaded symbols
regardless of regime) is directly addressable without memtrace —
the structural problem is visible from code inspection, not OCaml
heap forensics. Stage 4.5 PR-A introduces a cheap-stage-classify
phase before the heavy `Stock_analysis.analyze_with_callbacks` so
per-symbol work is proportional to **candidate** count (typically
≪ loaded count), not loaded count.

If Stage 4.5 PR-A doesn't drop β as expected, **then** memtrace per
the post-D spike note's recommendation.
